# 智能提醒去知识库但保留向量检索：实施方案

## 0. 先复述问题

当前智能提醒/智能推荐链路已经从“外部知识库检索”演进为“本项目内图片向量检索”：

- 推荐入口：`cloudfunctions/smartRecommendPhoto/index.js`
  - 生成用户需求文本向量：`createTextEmbedding(...)`
  - 读取本地集合：`clothes_image_vectors`
  - 用 `pickTopKBySimilarity(...)` 做余弦相似度 TopK
  - 再把命中的衣物图片传给多模态模型生成推荐文案
- 衣物向量构建入口：`cloudfunctions/rebuildUserKnowledgeBase/index.js`
  - 实际上现在已经不是“外部知识库重建”，而是在为衣物图片生成 embedding，并写入 `clothes_image_vectors`
- 新增/更新衣物入口：
  - `cloudfunctions/addClothing/index.js`
  - `cloudfunctions/updateClothing/index.js`
  - 目前仍保留大量 `knowledge_sync_*`、`bailian_doc_id`、`knowledge_doc_id` 等字段和命名，并通过 `triggerKnowledgeSyncInBackground` 调 `rebuildUserKnowledgeBase`

因此用户提出的核心诉求是：

1. **删掉“外部知识库”依赖与字段**，因为后续外部知识库会产生费用。
2. **不能影响现在的向量检索**，也就是必须保留 `clothes_image_vectors`、图片 embedding、文本 embedding、相似度排序、推荐降级逻辑。
3. **后续应在导入/新增衣服时就生成并存储衣服向量值**，而不是依赖用户手动点击“补同步/强制全量重同步”。
4. **强制按钮用处不大但可以保留**，它应该成为“向量补建/强制重建”调试入口，而不是“知识库重建”入口。

本方案只写计划，不直接改代码；实际代码任务由另一个 agent 执行。

## 1. 当前项目实际情况梳理

### 1.1 当前有效链路

#### `smartRecommendPhoto/index.js`

当前推荐主链路已经依赖向量集合：

```js
const { createTextEmbedding, requestCompatibleChat } = require('./common/dashscope-multimodal-provider')
const { pickTopKBySimilarity } = require('./common/image-vector-utils')

async function loadUserImageVectors(userId) {
  const res = await db.collection('clothes_image_vectors')
    .where({ user_id: userId })
    .limit(100)
    .get()
  return (res.data || []).filter(item => Array.isArray(item.vector) && item.vector.length)
}
```

这部分必须保留。所谓“删知识库”不能删除：

- `clothes_image_vectors`
- `createTextEmbedding`
- `createImageEmbedding`
- `pickTopKBySimilarity`
- `image_embedding_status`
- `image_embedding_error`
- `image_embedding_dim`
- `image_embedding_updated_at`

#### `rebuildUserKnowledgeBase/index.js`

虽然函数名叫 `rebuildUserKnowledgeBase`，但当前实现已经是图片向量重建：

```js
const vector = await createImageEmbedding({ imageUrl, apiKey, model, timeoutMs })
await db.collection('clothes_image_vectors').add({ data: { user_id, clothing_id, vector, vector_dim } })
await markClothingReady(clothing._id, vector.length)
```

因此本轮不建议立刻删除该云函数入口，避免前端按钮、云函数配置、已有调用全部失效。建议先把它“语义改造”为 `rebuildUserImageVectors`，但对外仍兼容旧函数名。

#### `addClothing/index.js` 与 `updateClothing/index.js`

这两个函数目前同时写入：

- 仍需要的图片向量字段：`image_embedding_status` 等
- 不再需要的外部知识库字段：`knowledge_sync_status`、`knowledge_doc_id`、`bailian_doc_id` 等

而且新增衣物后目前调用：

```js
triggerKnowledgeSyncInBackground({ cloud, openid, limit: 5 })
```

这个后台触发实际会调用 `rebuildUserKnowledgeBase`，但只按用户批量扫描，不够精确。后续应改成：新增/更新某一件衣物后，只为这件衣物生成/更新向量。

## 2. 总体设计原则

1. **只删除外部知识库，不删除本地向量库**。
   - 删除/停用 `knowledge_sync_*`、`bailian_*`、`knowledge_doc_id` 语义。
   - 保留 `image_embedding_*` 与 `clothes_image_vectors`。
2. **兼容旧入口，渐进改名**。
   - 保留 `rebuildUserKnowledgeBase` 云函数，避免页面按钮和云函数配置失效。
   - 文案、日志、返回 message 改成“图片向量”。
3. **新增/更新衣物即生成向量**。
   - 新增一个共享向量同步模块，支持按 `clothingId` 精准同步。
   - `addClothing` 成功写库后后台触发单件向量生成。
   - `updateClothing` 图片或关键元信息变化时后台触发单件向量更新。
4. **手动按钮保留为维护入口**。
   - “补同步/查状态”：补建缺失/失败的向量。
   - “强制全量重同步”：强制重建全部向量。
5. **日志只在异常/警告时输出**。
   - 成功路径不刷大量 debug。
   - 向量维度不一致、空向量、临时 URL 失败、API 失败、DB 写失败必须 `logWarning` 或 `logError`。
6. **每个子任务都基于现有架构迭代，不重做一套系统**。

function hasVectorRelevantChanges(current = {}, nextPayload = {}) {
  return !isSameString(current.image, nextPayload.image)
    || !isSameString(current.originalImage, nextPayload.originalImage)
    || !isSameString(current.name, nextPayload.name)
    || !isSameString(current.season, nextPayload.season)
    || !isSameString(current.category, nextPayload.category)
    || !isSameTagList(current.tags, nextPayload.tags)
    || !isSameString(current.material, nextPayload.material)

function buildImageEmbeddingResetFields(nextPayload = {}) {

# 子任务 2：新增单件图片向量同步能力，导入衣服时自动写入向量

## 1. 任务目标

实现“上传/导入衣服时就存好衣服向量值”：新增或更新某件衣物后，后台精准生成该衣物图片 embedding，写入 `clothes_image_vectors`。

## 2. 实现方案

### 2.1 抽取共享向量同步服务

建议新增：

```text
cloudfunctions/common/image-vector-sync-service.js
```

但由于微信云函数部署时通常只打包各自目录，实际更稳妥的第一阶段做法是：

- 在 `cloudfunctions/rebuildUserKnowledgeBase/` 内实现 `syncSingleClothingVector`
- `addClothing` / `updateClothing` 通过 `cloud.callFunction` 调 `rebuildUserKnowledgeBase`，传 `clothingId`

修改 `cloudfunctions/rebuildUserKnowledgeBase/index.js` 支持：

```js
const clothingId = normalizeText(event.clothingId)
```

加载衣物逻辑：

```js
async function loadSyncableClothes({ userId, clothingId, limit, forceResync }) {
  let query = db.collection('clothes').where({ user_id: userId })

  if (clothingId) {
    const res = await db.collection('clothes')
      .where({ _id: clothingId, user_id: userId })
      .limit(1)
      .get()
    return (res.data || []).filter(item => shouldSyncClothing(item, true))
  }

  const res = await query
    .orderBy('created_at', 'desc')
    .limit(Math.min(100, Math.max(limit * 2, limit)))
    .get()

  return (res.data || [])
    .filter(item => shouldSyncClothing(item, forceResync))
    .slice(0, limit)
}
```

注意：单件同步时 `shouldSyncClothing(item, true)`，确保新增后立即执行，不受旧状态影响。

### 2.2 新增后台触发器

新增 `cloudfunctions/addClothing/utils/image-vector-sync-trigger.js`，`updateClothing` 可复制一份或后续抽公共包：

```js
function normalizeText(value) {
  return typeof value === 'string' ? value.trim() : ''
}

function logWarning(scope, message, extra = {}) {
  console.warn(`[ImageVectorSyncTrigger][${scope}] ${message}`, extra)
}

function triggerImageVectorSyncInBackground({ cloud, openid, clothingId, forceResync = true } = {}) {
  const targetOpenid = normalizeText(openid)
  const targetClothingId = normalizeText(clothingId)

  if (!cloud || typeof cloud.callFunction !== 'function') {
    logWarning('trigger', 'cloud.callFunction unavailable')
    return
  }
  if (!targetOpenid || !targetClothingId) {
    logWarning('trigger', 'missing openid or clothingId', { hasOpenid: Boolean(targetOpenid), hasClothingId: Boolean(targetClothingId) })
    return
  }

  Promise.resolve()
    .then(() => cloud.callFunction({
      name: 'rebuildUserKnowledgeBase',
      data: {
        targetOpenid,
        clothingId: targetClothingId,
        limit: 1,
        forceResync,
        mode: 'image_vector'
      }
    }))
    .catch(error => {
      logWarning('trigger', 'image vector sync trigger failed', {
        clothingId: targetClothingId,
        errMsg: error && error.message ? error.message : String(error)
      })
    })
}

module.exports = { triggerImageVectorSyncInBackground }
```

- `vector_dim === vector.length`。

## 4. 验证成功方法

1. 新增一件带图衣物后，不点按钮，等待数秒：
   - `clothes.image_embedding_status` 变为 `ready`
   - `clothes.image_embedding_dim > 0`

继续复用现有 `rebuildUserKnowledgeBase` 的向量生成能力，只增加 `clothingId` 精准同步能力，不另建一套独立云函数。

# 子任务 3：把 `rebuildUserKnowledgeBase` 语义改造成“图片向量维护”，保留旧入口兼容

function buildVectorSyncStatsSummary(stats = {}) {
  const total = normalizeCount(stats.total)
  const readyCount = normalizeCount(stats.readyCount)
  const failedCount = normalizeCount(stats.failedCount)
  const skippedCount = normalizeCount(stats.skippedCount)

  let status = 'idle'
  if (readyCount > 0 && failedCount === 0) status = 'success'
  else if (readyCount > 0 && failedCount > 0) status = 'partial_success'
  else if (failedCount > 0) status = 'failed'
  else if (skippedCount > 0) status = 'skipped'

  return {
    status,
    text: `共处理 ${total} 件，成功生成向量 ${readyCount} 件，失败 ${failedCount} 件，跳过 ${skippedCount} 件。`
  }
}

module.exports = {
  normalizeLimit,
  buildVectorSyncStatsSummary,
  buildRebuildStatsSummary: buildVectorSyncStatsSummary
}

# 子任务 6：强化向量检索的数据获取与输出验证

## 2. 实现方案

### 2.1 修改 `smartRecommendPhoto/index.js` 的 `loadUserImageVectors`

当前只过滤 `Array.isArray(item.vector) && item.vector.length`，建议更严格：

```js
const { isValidVector, pickTopKBySimilarity } = require('./common/image-vector-utils')
```

```js
async function loadUserImageVectors(userId) {
  try {
    const res = await db.collection('clothes_image_vectors')
      .where({ user_id: userId })
      .limit(100)
      .get()

    const docs = res.data || []
    const validDocs = docs.filter(item => isValidVector(item.vector))
    const invalidCount = docs.length - validDocs.length

    if (invalidCount > 0) {
      logWarning('recommend.imageRetrieval', 'invalid image vectors skipped', {
        total: docs.length,
        invalidCount
      })
    }

    return validDocs
  } catch (error) {
    const message = error && error.message ? error.message : String(error)
    if (/collection not exists|Db or Table not exist|DATABASE_COLLECTION_NOT_EXIST|ResourceNotFound/i.test(message)) {
      logWarning('recommend.imageRetrieval', 'image vector collection not found')
      return []
    }
    throw error
  }
}
```

### 2.2 捕获维度不一致导致的空命中

`pickTopKBySimilarity` 当前会过滤维度不一致。建议在调用前计算维度分布：

```js
function summarizeVectorDocs(vectorDocs = [], queryVector = []) {
  const queryDim = Array.isArray(queryVector) ? queryVector.length : 0
  const total = vectorDocs.length
  const sameDimCount = vectorDocs.filter(doc => Array.isArray(doc.vector) && doc.vector.length === queryDim).length
  return {
    queryDim,
    total,
    sameDimCount,
    skippedByDimCount: Math.max(0, total - sameDimCount)
  }
}
```

调用：

```js
const vectorStats = summarizeVectorDocs(vectorDocs, queryVector)
if (vectorStats.skippedByDimCount > 0) {
  logWarning('recommend.imageRetrieval', 'image vectors skipped by dimension mismatch', vectorStats)
}
```

const recommendation = await buildMultimodalRecommendation({ ... })
