# 智能提醒后端切换阿里百炼知识库实施方案

## 1. 问题复述

当前项目已经完成“智能推荐 / 智能提醒”页面与试穿页的主流程 UI，但后端知识库链路当前基于智谱实现，实际运行中已经暴露出稳定性问题：

1. 当前知识库虽然能创建成功，但文件进入知识库后长期处于“数据异常”；
2. 即使执行了批量补同步，知识库中文件也无法切片，导致 `retrievalHitCount = 0`；
3. 旧链路已不再依赖千帆，当前主阻塞点已经集中到“知识库平台能力 + 文件入库 + 检索返回”；
4. 用户希望将后端改造成 **阿里百炼知识库版**，并保留现有页面、数据库与试穿页逻辑。

因此，本次目标不是重做整个系统，而是：

```text
在当前项目架构基础上
将智能推荐后端从“智谱知识库实现”切换为“阿里百炼知识库实现”
并继续保证：
用户输入需求 → 检索衣橱 → 选出衣物 → 返回 selectedClothesIds → 跳试穿页
```

---

## 2. 结合当前项目的真实现状

以下内容基于当前仓库代码，而不是理想状态。

### 2.1 当前后端入口

现有主云函数：

- `cloudfunctions/smartRecommendPhoto/index.js`

当前已经演进到“知识库优先”结构，核心链路大致是：

```text
读取当前用户
→ 读取 users.knowledge_id
→ 查询 clothes
→ 同步 pending 衣物到知识库
→ 对 knowledge_id 发起检索
→ 将命中的 doc_id 映射回 clothes._id
→ 返回 selectedClothesIds
```

### 2.2 当前数据库结构已为知识库链路做过扩展

#### users 集合已有/应已有字段

```js
{
  knowledge_id,
  knowledge_status,
  knowledge_provider,
  knowledge_bound_at
}
```

#### clothes 集合已有/应已有字段

```js
{
  image,
  originalImage,
  retrieval_tags,
  retrieval_text,
  knowledge_doc_id,
  knowledge_sync_status,
  knowledge_sync_error,
  knowledge_last_sync_at
}
```

这意味着：

- 本地业务主数据仍然是 `clothes`
- 知识库只是检索增强层
- 当前试穿页仍然依赖 `selectedClothesIds`

### 2.3 当前试穿页契约不能变

`pages/tryon/tryon.helpers.js` 仍然以：

```js
selectedClothesIds
```

为入口做自动选衣和摆放。

因此，不管底层换成智谱、阿里百炼还是别的平台，都必须保证：

```text
检索结果最终可映射回 clothes._id
```

### 2.4 当前已知失败事实

从前面的真实调试结果可确认：

1. 智谱知识库已经能创建；
2. 文件已经能“进入知识库列表”；
3. 但文件状态长期为“数据异常”；
4. 检索结果长期为空；
5. 现有 Zhipu 实现即使改成文件上传，仍未跑通。

因此，本次切换阿里百炼并不是“锦上添花”，而是当前主路线调整。

## 3. 阿里百炼官方资料结论（基于官方文档）

以下为本方案依赖的官方资料与我据此得出的工程结论。

### 3.1 官方能力结论

#### 结论 A：百炼知识库支持图片文件

阿里百炼官方文档说明：

- 文档搜索类知识库支持：
  - `png`
  - `jpg/jpeg`
  - `bmp`
  - `gif`

并给出了图片尺寸与大小限制。

参考：
- 知识库配额与限制  
  <https://www.alibabacloud.com/help/zh/model-studio/rag-knowledge-base-specifications>

#### 结论 B：百炼“文档搜索”类型支持视觉理解场景

官方文档说明：

- 创建知识库时，`文档搜索` 类型下可选：
  - 基础文档问答
  - 图文并茂回复
  - **视觉理解（富文本文档）**
  - 极速问答
- 其中“视觉理解（富文本文档）”会自动切换到：
  - `qwen3-vl-embedding`
- 该场景支持：
  - PDF
  - 图片
  - 图文组合命中测试

参考：
- 知识库  
  <https://help.aliyun.com/zh/model-studio/rag-knowledge-base>

#### 结论 C：对外检索推荐走 `file_search`

官方文档说明：

- 百炼知识检索可通过 Responses API 调用；
- 在 `tools` 中添加 `file_search`；
- 通过 `vector_store_ids` 指定知识库 ID；
- 当前 `vector_store_ids` 仅支持一个知识库 ID。

参考：
- 知识检索  
  <https://help.aliyun.com/zh/model-studio/file-search>

#### 结论 D：知识库管理 API 与 DashScope API 不是一套鉴权

官方文档说明：

- 百炼知识库管理 API 推荐通过阿里云 OpenAPI / SDK 调用；
- 需要：
  - `ALIBABA_CLOUD_ACCESS_KEY_ID`
  - `ALIBABA_CLOUD_ACCESS_KEY_SECRET`
  - `WORKSPACE_ID`
  - RAM / 业务空间权限
- 而知识检索 `file_search` 则是另一条调用路径，通常走：
  - `DASHSCOPE_API_KEY`

参考：
- 知识库 API 指南  
  <https://help.aliyun.com/zh/model-studio/rag-knowledge-base-api-guide>

#### 结论 E：知识库创建后类型不可更改

这意味着：

- 不能先创建“基础文本库”再改成“视觉理解”；
- 必须从创建阶段就选对知识库类型。

参考：
- 知识库  
  <https://help.aliyun.com/zh/model-studio/rag-knowledge-base>

### 3.2 对当前项目的直接影响

基于上述文档，切换百炼时必须接受两个现实：

1. **知识库管理与知识检索要分层实现**
   - 管理侧：AK/SK + WORKSPACE_ID
   - 检索侧：DASHSCOPE_API_KEY

2. **不能继续沿用“只靠一个 provider API key 就包打天下”的设计**

这决定了本次切换不是简单替换 URL，而是：

```text
保留当前项目结果结构
替换知识库 provider 与同步链路
并补充阿里云管理鉴权层
```

## 子任务 1：先把 Provider 配置层从 Zhipu 固化改成 Bailian 双鉴权配置层

### 1. 任务目标

将当前 `smartRecommendPhoto` 中以 `ZHIPU_API_KEY` 为核心的 provider 配置，改造成适配阿里百炼的双鉴权配置层。

### 2. 实现方案

新增配置模块：

```js
// cloudfunctions/smartRecommendPhoto/utils/bailian-config.js
const { logWarning } = require('./logger')

let localConfig = {}
try {
  localConfig = require('../config.local')
} catch (error) {
  localConfig = {}
}

function normalizeText(value) {
  return typeof value === 'string' ? value.trim() : ''
}

function getBailianConfig() {
  return {
    dashscopeApiKey: process.env.DASHSCOPE_API_KEY || localConfig.DASHSCOPE_API_KEY || '',
    accessKeyId: process.env.ALIBABA_CLOUD_ACCESS_KEY_ID || localConfig.ALIBABA_CLOUD_ACCESS_KEY_ID || '',
    accessKeySecret: process.env.ALIBABA_CLOUD_ACCESS_KEY_SECRET || localConfig.ALIBABA_CLOUD_ACCESS_KEY_SECRET || '',
    workspaceId: process.env.WORKSPACE_ID || localConfig.WORKSPACE_ID || '',
    region: process.env.BAILIAN_REGION || localConfig.BAILIAN_REGION || 'cn-beijing'
  }
}

function validateBailianManagementConfig(config = getBailianConfig()) {
  const valid = Boolean(
    normalizeText(config.accessKeyId)
    && normalizeText(config.accessKeySecret)
    && normalizeText(config.workspaceId)
  )

  if (!valid) {
    logWarning('bailianConfig.validateManagement', 'missing ak/sk/workspace config')
  }

  return valid
}

function validateBailianSearchConfig(config = getBailianConfig()) {
  const valid = Boolean(normalizeText(config.dashscopeApiKey))
  if (!valid) {
    logWarning('bailianConfig.validateSearch', 'missing dashscope api key')
  }
  return valid
}
```

同时把当前：

```js

```

```js

```

但先不切主逻辑，只做配置与 provider 注册准备。

## 子任务 3：改造衣物同步链路，切到 Bailian 官方“文件导入 + 索引作业”流程

### 1. 任务目标

把当前基于 Zhipu 上传的同步服务，改成符合百炼官方文档的文件导入流程，并确保同步状态可追踪。

### 2. 实现方案

根据百炼知识库 API 指南，建议按官方步骤拆成：

```text
1. ApplyFileUploadLease
2. 上传文件到租约 URL
3. AddFile / 关联类目
4. SubmitIndexAddDocumentsJob
5. GetIndexJobStatus 轮询直到完成
6. 回写 bailian_file_id / bailian_doc_id / knowledge_sync_status
```

建议新增：

```js
// cloudfunctions/smartRecommendPhoto/utils/bailian-sync-service.js
async function syncClothingToBailian({
  cloud,
  db,
  workspaceId,
  knowledgeId,
  clothing,
  provider
}) {
  if (!clothing || !clothing.image) {
    logWarning('bailianSyncService.syncClothingToBailian', 'missing clothing image', {
      clothingId: clothing && clothing._id
    })
    return { status: 'failed', reason: 'missing image' }
  }

  const download = await cloud.downloadFile({ fileID: clothing.image })
  const fileBuffer = download && download.fileContent
  if (!Buffer.isBuffer(fileBuffer) || !fileBuffer.length) {
    logError('bailianSyncService.syncClothingToBailian', new Error('empty file buffer'), {
      clothingId: clothing._id
    })
    return { status: 'failed', reason: 'empty file buffer' }
  }

  const lease = await provider.applyFileUploadLease({
    workspaceId,
    fileName: buildClothingFileName(clothing),
    fileMd5: calculateMd5(fileBuffer),
    fileSize: fileBuffer.length
  })

  await provider.uploadLeaseFile({
    uploadUrl: lease.uploadUrl,
    headers: lease.headers,
    fileBuffer
  })

  const fileEntry = await provider.addFileToKnowledge({
    workspaceId,
    knowledgeId,
    categoryId: 'default',
    fileUploadLeaseId: lease.leaseId
  })

  const job = await provider.submitIndexAddDocumentsJob({
    workspaceId,
    knowledgeId,
    fileIds: [fileEntry.fileId]
  })

  const finalStatus = await provider.waitIndexJobDone({
    workspaceId,
    knowledgeId,
    jobId: job.jobId
  })

  if (finalStatus !== 'success') {
    throw new Error(`index job failed: ${finalStatus}`)
  }

  await db.collection('clothes').doc(clothing._id).update({
    data: {
      bailian_file_id: fileEntry.fileId,
      bailian_doc_id: fileEntry.fileId,
      knowledge_sync_provider: 'bailian',
      knowledge_sync_status: 'ready',
      knowledge_sync_error: '',
      knowledge_last_sync_at: db.serverDate(),
      updated_at: db.serverDate()
    }
  })
}
```

### 3. 计算正确性、异常处理与单元测试

涉及计算：

- MD5
- 文件大小
- 轮询超时
- 同步统计

要求：

- fileBuffer 为空必须 `logError`
- 轮询超过阈值必须 `logWarning`
- `fileIds.length !== 提交数量` 时必须报错


- 控制台里知识库文件状态从“待处理”变为正常可用；

### 2. 实现方案

建议在 `clothes` 中优先维护：

```js
{
  bailian_file_id,
  knowledge_sync_status,
  _id
}
```

命中结果映射逻辑：

```js
function mapBailianFilesToClothesIds(searchResults = [], clothesList = []) {
  const fileToClothes = new Map(
    clothesList
      .filter(item => item && item.bailian_file_id && item._id)
      .map(item => [String(item.bailian_file_id), String(item._id)])
  )

  return [...new Set(
    searchResults
      .map(item => fileToClothes.get(String(item.fileId || '')))
      .filter(Boolean)
  )]
}
```
